Why Keyword Search Fails

"How much does it cost to fix my car?"

Your document says

"Automobile repair pricing starts at £80 per hour."

Search typeMatch?Why
Keyword ("car" in text)✗ MissZero shared words: car≠automobile, fix≠repair, cost≠pricing
Semantic (embeddings)✓ HitThe meanings are nearly identical

Synonyms, typos, paraphrases, other languages — keyword search misses them all.

What Is an Embedding?

An embedding is a fixed-length list of numbers that represents the meaning of a piece of text.

"I love pizza"      →  [ 0.12, -0.87, 0.44, ..., 0.03]   # 384 numbers
"Pizza is amazing"  →  [ 0.10, -0.85, 0.47, ..., 0.05]   # nearly the same!
"Tax law reform"    →  [-0.71,  0.22, -0.90, ..., 0.66]   # far away
  • Same size for any input — a word, a sentence, a whole chunk → always 384 numbers
  • Similar meaning → nearby vectors — that's the property everything else builds on
  • Produced by a neural network trained on billions of sentence pairs

Meaning Is Geometry

Food texts cluster together
Sports texts cluster together
Python texts cluster together
A query lands inside the cluster that answers it

Real embeddings live in 384-D, not 2-D — same idea, more room for nuance.

EraApproachLimitation

How Close Are Two Vectors?

MetricFormulaIntuition
Dot productΣ aᵢ·bᵢAlignment × magnitude
Cosine similaritya·b / (‖a‖‖b‖)Angle only — ignores length. 1 = same direction, 0 = unrelated
Euclidean (L2)‖a − b‖Straight-line distance — smaller = closer

Cosine is the default for text. Bonus: if you normalize vectors to length 1, cosine = dot product — vector DBs exploit this for speed.

Cosine by Hand — 2-D Warm-up

a = [3, 4]      # "pizza"
b = [4, 3]      # "pasta"
c = [-4, 3]     # "tax law"

dot(a,b) = 3·4 + 4·3 = 24
‖a‖ = 5,  ‖b‖ = 5

cos(a,b) = 24 / 25 = 0.96   # close!
cos(a,c) = 0 / 25  = 0.00   # unrelated

In NumPy

import numpy as np

def cosine(a, b):
    return np.dot(a, b) / (
        np.linalg.norm(a) *
        np.linalg.norm(b)
    )

You'll compute one by hand in the notebook before NumPy does it for you.

Notebook · Section 6

Semantic Search in 15 Lines

corpus_embs = model.encode(chunks, normalize_embeddings=True)   # (N, 384)

def search(query: str, k: int = 3):
    q = model.encode(query, normalize_embeddings=True)          # (384,)
    scores = corpus_embs @ q                                    # (N,) cosine
    top = np.argsort(scores)[::-1][:k]
    return [(chunks[i], float(scores[i])) for i in top]

for text, score in search("footballer who scored in the final"):
    print(f"{score:.3f}  {text[:80]}")

Embed once, search many times. This is the heart of every RAG system — everything after this is scaling and plumbing.

What Embeddings Are Bad At

WeaknessExampleMitigation
Negation"pizza is great" ≈ "pizza is NOT great"LLM re-reads retrieved text anyway
Exact identifiersOrder #48291, SKU codes, namesKeyword/hybrid search (Week 4)
Numbers & math"under £50" won't filter pricesMetadata filters (Class 3)
FreshnessModel froze at training timeYou embed your current docs — that's the point of RAG

Similar direction ≠ same truth value. Embeddings find relevant text; the LLM still does the reading.
